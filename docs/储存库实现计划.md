# 储存库（Storage Vault）实现计划

> 状态：计划（未开工）  
> 范围：`Addons/Logistics` — 物流存储单元 + 共享储存库  
> 调研基准：宿主 `Survivalcraft/Component/ComponentInventoryBase.cs`、IE2 流体罐 / LED 屏连通组、Logistics 现有方块与滚动面板

---

## 1. 目标与术语

| 术语 | 定义 |
| --- | --- |
| **储存单元** | 方块 `LogisticsStorageUnitBlock`（Index 552，写法同 SCIENEW 普通方块）。每块贡献 **24** 个槽位容量；可交互；对外暴露 `IInventory`。 |
| **储存库** | 一堆**六向连通**的储存单元共享的逻辑库存。容量 = `单元数 × 24`。内容物**紧密堆叠**（无「中间空槽」）。 |
| **储存库堆 / 簇** | 一次 flood-fill 得到的连通块集合；一簇对应唯一一个储存库实例。 |

### 1.1 必须满足

1. 储存库 ≈ 大号 `ComponentInventoryBase` 的槽列表语义，但**不实现** `IInventory`。
2. 可随时扩容 / 缩容；槽数随簇大小动态变化。
3. 紧密堆叠：不会出现 `[A] [空] [B]`；空槽只允许出现在「已用前缀」之后。
4. 同簇单元共享同一储存库。
5. 每个储存单元向游戏暴露 `IInventory`，**转发**到所属储存库，供抓取机等读写。
6. 点击任一单元 → 打开**该簇储存库** UI；格子多时**分页**（Grid 放不下时）。
7. 加入单元 → 容量 +24；移除单元 → 容量 −24；若缩容导致物品无处安放，在**被移除单元的世界坐标**爆出。

### 1.2 非目标（本阶段不做）

- 跨空格「无线」组网、频道、优先级（AE2 / RS 网络语义）。
- 流体储存、过滤分区、合成终端。
- 与主仓 Factorio 传送带共用库存模型。
- 簇超过 64 后的自动拆分 / 桥接块（超限则拒并入，见 Q4）。

## 2. 调研摘要

### 2.1 宿主库存（硬约束）

路径：`Survivalcraft/Game/IInventory.cs`、`Component/ComponentInventoryBase.cs`。

| 点 | 结论 |
| --- | --- |
| 槽结构 | `List<Slot>`，`Slot { Value, Count }`；`SlotsCount => m_slots.Count` |
| 装入 | `FindAcquireSlotForItem`：先同种未满，再**第一个空槽**（不压缩） |
| `AddSlotItems` | 种类不符或超容量 → 抛异常 |
| `RemoveSlotItems` | 减 Count，**不保证清空 Value** |
| Load | 从模板读固定 `SlotsCount`，再建 `Slot{i}` |
| Save | **不写** `SlotsCount`，只写非空槽 |

→ 动态槽数必须**自写 Load/Save 的 `SlotsCount`**，不能只靠 xdb 模板常数。  
→ 紧密堆叠宿主不提供，必须自写 `Compact` / 装入路径。

### 2.2 打开 UI / 方块实体

- 原版：`SubsystemChestBlockBehavior` → `ModalPanelWidget = new ChestWidget(...)`；移除前 `DropAllItems`。
- IE2：`EntityDeviceBehavior` + `IEntityDevice.GetWidget`；`BlockEntityUtils.Create/RemoveBlockEntity`（Remove 同样会 Drop）。

**陷阱**：拓扑重建时若 `RemoveBlockEntity` 仍挂着满库存的 `IInventory`，会整库洒落。储存库迁移必须**先搬物品 / 解绑 Drop，再删实体**。

### 2.3 IE2 多方块对照

| 模式 | 代表 | 可借鉴 | 不可照搬 |
| --- | --- | --- | --- |
| 底格锚点 + 容量伸缩 | 流体罐 `ComponentFluidTank.UpdateVolume` | 「块数 → 容量」 | 单体积标量，非多槽 |
| Flood-fill + 词典序锚点 | `LEDScreenController` / `SubsystemLEDScreenControllers` | 选锚点、组重建 | 重建常删建实体 → 库存场景危险 |
| Subsystem 字典 | 太阳能板组 | 拓扑表 | 库存仍应落在 Component 存档 |

外部参考（概念，非代码移植）：AE2 Drive / RS Grid 是**逻辑网络共享库存**；本需求是**物理邻接多方块共享一个稠密槽表**，更接近「可堆叠的大箱子」而非 ME 网络。

### 2.4 Logistics 现状

- `LogisticsStorageUnitBlock`：仅立方体渲染，无 Behavior / Entity / 库存。
- 抓取机找邻库存应复用 IE2 `ComponentInserter`：`GetBlockEntity` → `FindComponent<IInventory>()`。
- `LogisticsScrollPanelWidget`：库存拖拽中不抢 Drag，适合大库存滚动；**分页**宿主无现成控件，需自做。

## 2.5 储存库归属：方案 A / B / C（选型）

| 代号 | 含义 |
| --- | --- |
| **A** | 本文初稿：词典序**锚点 Entity** 挂 `ComponentStorageVault`；成员只转发到锚点。 |
| **B** | 全局 Subsystem：`Guid → 储存库`；单元只存 Guid；建簇时 `new Guid` + 实例化，并入时写入已有 Guid。 |
| **C** | Minecraft 同类模组的主流做法（见下，均有公开信源）。 |

### C 的可信对照（非臆造）

1. **Refined Storage（最接近 B）**  
   - 官方 CHANGELOG：[refinedmods/refinedstorage CHANGELOG](https://github.com/refinedmods/refinedstorage/blob/develop/CHANGELOG.md) 原文：  
     *「storage disk data (items, fluids) are stored off the disk itself, in another file (`refinedstorage_disks.dat`). The disk itself only stores its ID.」*  
   - 官方 Javadoc：`StorageDiskManager` 以 `UUID → IStorageDisk` 注册，走世界 `SavedData`（[StorageDiskManager](https://refinedmods.com/javadoc/refinedstorage/com/refinedmods/refinedstorage/apiimpl/storage/disk/StorageDiskManager.html)）。  
   - 后果有据：磁盘内容损坏时需恢复 `refinedstorage_disks.dat`（例：[Issue #3714](https://github.com/refinedmods/refinedstorage/issues/3714)）。

2. **Applied Energistics 2（网络聚合，不是「一堆方块一个袋子」）**  
   - 官方 [API.md](https://github.com/AppliedEnergistics/Applied-Energistics-2/blob/main/API.md)：Grid 由 **BlockEntity/Part 持有的 GridNode** 自动连接形成；节点通过 `IStorageProvider.mountInventories` 把**各自**库存挂进网络存储服务。  
   - 物品仍在 Drive/Cell 等节点侧；网络是运行时索引，不是「簇共享一个 Guid 库」。

3. **Storage Drawers（更接近「门面聚合」）**  
   - CurseForge 说明：[Storage Drawers](https://www.curseforge.com/minecraft/mc-mods/storage-drawers) — Drawer Controller 在范围内搜索抽屉，**暴露为大合并库存**；各抽屉仍自有库存。Controller Slave 只是额外 IO 面（作者在 [Issue #482](https://github.com/jaquadro/StorageDrawers/issues/482) 澄清不扩范围）。

→ **C 的结论**：真正「引用 ID、内容在全局表」的成熟先例是 **RS 磁盘 UUID**；物理邻接多方块更常见的是 **每块自有库存 + Controller 聚合（Drawers）** 或 **成型多方块 Master BE（IE 等）**。你们要的是「邻接共享**一个**稠密库」，语义上更接近 **RS 的 Guid 库** + **Drawers 的邻接组网**，而不是 AE2 全网络。

### A vs B 在 Survivalcraft 下

| 维度 | A 锚点 Component | B Subsystem + Guid |
| --- | --- | --- |
| 存档落点 | Entity Component（与箱子同构） | Subsystem Load/Save（合规，见 world-save-persistence） |
| 跨区块簇 | 锚点区块卸载 → 成员格可能找不到 Vault | 库在 Project 级，不绑单格 Entity |
| 并入 / 挖断 / 迁锚 | 必须搬库存、防 `RemoveBlockEntity` 双 Drop | 改 Guid 引用 + 合并/拆分字典项，无「库存 Entity 搬家」 |
| 孤儿数据 | 少（随锚点 Entity 生死） | 需 GC：无引用 Guid 时删除或清空库 |
| 与 IE2 现有模式 | 像 LED 锚点 / 流体罐底格 | 像 TechTree 等「世界一份」Subsystem，也像 RS |
| 抓取机 | 仍要每格 `IInventory` 代理 | 同左：代理 `Guid → Subsystem.Get(Guid)` |

### 选型结论（推荐 **B**）

在 SC 里若做「墙式多格存储、可跨区块、频繁拆合并」，**B 更合适**：避免 A 的锚点迁移与卸载陷阱；SC 的 `Subsystem` Save 正好对应 RS 的 SavedData。  
采用 B 时必须带上 C（RS）的教训：**Guid 生命周期与 GC 写清楚**，合并只保留一个 Guid，拆分给新分量 `new Guid` 并切物品，簇空则 `Remove(Guid)`，禁止无限堆积幽灵库。

**A 仍可作为「单区块、极简原型」**，但不应作为终局归属。

## 3. 架构总览（方案 B）

```text
        SubsystemStorageVaults  (Project 级)
        ┌──────────────────────────────────────────┐
        │  Dictionary<Guid, StorageVault>          │
        │  StorageVault: 稠密槽表, Capacity=N*24   │
        │  不实现 IInventory；自写 Load/Save        │
        └──────────────▲───────────────────────────┘
                       │ Get(Guid)
         ┌─────────────┼─────────────┐
         │             │             │
┌────────┴───┐  ┌──────┴────┐  ┌─────┴──────┐
│ Unit Entity│  │ Unit      │  │ Unit       │
│ 存 Guid    │  │ 同 Guid   │  │ 同 Guid    │
│ IInventory │  │ IInventory│  │ IInventory │
│ 转发→Vault │  │ 转发      │  │ 转发       │
└────────────┘  └───────────┘  └────────────┘
```

### 3.1 为何「每格都有实体 + IInventory 代理」

抓取机路径是 `邻格 BlockEntity → FindComponent<IInventory>`。  
因此：**每个储存单元都有 BlockEntity**；`ComponentStorageUnit` 实现 `IInventory`，用本地 `VaultGuid` 向 Subsystem 取库并转发。

### 3.2 Guid 生命周期

| 事件 | 行为 |
| --- | --- |
| 孤放置第一格 | `Guid.NewGuid()`，`Subsystem.CreateVault(guid)`，单元写入 Guid |
| 邻接已有簇 | 若簇已达 **64** 单元：拒绝并入（本格自成新 Guid 或放置失败策略见 Behavior）；否则单元写入已有 Guid，`Vault.Resize(+24)` |
| 两簇合并 | 合并后若总块数 > 64：不合并，保持两 Guid；否则选保留 Guid，合并槽表 Compact，全体改 Guid，`Remove` 被吞并 Guid |
| 移除一格 | `Resize(-24)`，溢出在移除点爆出；若簇空则 `Remove(Guid)` |
| 挖断分裂 | 一分量保留旧 Guid；其它分量 `NewGuid` + **Q1 槽序切分**物品 |

### 3.3 子系统职责 `SubsystemStorageVaults`

| 职责 | 说明 |
| --- | --- |
| `Dictionary<Guid, StorageVault>` | 权威库存；Load/Save 整表 |
| `CollectCluster` / `RebuildAt` | flood-fill 与拓扑事件 |
| `Get(Guid)` / `Create` / `Remove` | Guid API |
| 可选 `cell → Guid` 缓存 | 加速；可重建，非存档真相 |
| 孤儿 GC | 重建后无任何 Unit 引用的 Guid → 删除（内容应已处理） |

须在 `Logistics.xdb` 的 `ProjectTemplate` 注册该 Subsystem。

## 4. 数据模型与紧密堆叠

### 4.1 `StorageVault`（核心，挂在 Subsystem 字典内，非 Component）

```text
Guid Id
常量
SlotsPerUnit = 24
List<(int Value, int Count)> m_slots
int MemberCount
int Capacity => MemberCount * SlotsPerUnit

不实现 IInventory
对外 API（建议）：
  TryInsert / TryExtract / RemoveAt / Compact / ResizeCapacity
  GetSlot*(...)  // 供 Unit 转发与 UI
序列化：由 Subsystem 写入 Guid + SlotsCount + 非空槽
```

**禁止**把 Vault 当 `IInventory` 暴露；内部可复刻 `ComponentInventoryBase` 槽算法并改紧密装入。

### 4.2 紧密不变式

任意时刻存在前缀长度 `Used`：

- `i < Used` → `Count > 0`
- `i ≥ Used` → 空（Capacity 内可保留空槽结构，或缩到 `Used` 并在扩容时再 Append）

推荐实现策略：

1. **逻辑容量** = `Capacity`（含尾部空槽，便于 `InventorySlotWidget` 固定 index）。
2. **Compact 时机（对齐宿主拖拽）**：`Add` / `Remove` 只 `RequestCompact`；由 `SubsystemStorageVaults.Update` **帧末** `CompactIfNeeded`。这样 `InventorySlotWidget` 的 Remove→Add 整笔完成后再压实，支持库内整理顺序；拓扑合并/拆分/存档/爆出仍可立刻 `Compact()`。
3. Compact **只去空洞、不重排相对顺序**（`[C][B][A]` 保持；`[A][空][B]` → `[A][B]`）。
4. `TryInsert` / 抓取机装入：同种合并 → 再填第一个空槽；帧末 Compact 收尾。

常量：`MaxUnitsPerCluster = 64`（**Q4**，= 1536 槽）。  
稳态下不得出现中间空洞；拖拽过程中允许短暂空洞。

### 4.3 `ComponentStorageUnit : Component, IInventory`

存档字段：`VaultGuid`。全部 `IInventory` 转发到 `SubsystemStorageVaults.Get(VaultGuid)`。  
Guid 暂时无效时读写返回空/失败。`DropAllItems` 默认 no-op（物品在 Subsystem）；爆出只走缩容/拆库逻辑。

## 5. 拓扑：放置 / 移除 / 合并 / 分裂

### 5.1 连通规则

- 六向邻接；`BlocksManager.Blocks[GetCellContents] is LogisticsStorageUnitBlock`（宿主惯用类型判定，兼容动态 Index）。
- 单格自成一簇（容量 24）。

### 5.2 放置 `OnBlockAdded`

```text
1. 创建本格 BlockEntity + ComponentStorageUnit；挂 gather 库存（Q5）
2. Collect 邻接簇
3. 若可并入唯一邻簇且并入后 ≤ 64：
   - 写入该 Guid，Resize(+24)
4. 若邻接多个簇且合并后 ≤ 64：
   - 选保留 Guid，合并槽表 Compact，全体改 Guid，Remove 其余 Guid，Resize
5. 若超 64 或无法并入：本格 NewGuid 自成一库（容量 24）
6. 刷新 cell→Guid；通知已打开 UI 重建
```

### 5.3 移除 `OnBlockRemoved`（用户指定行为）

```text
1. 记录移除点 worldPos、旧簇成员、旧 Vault 快照引用
2. 先从簇中拿掉该格，得到 0..N 个剩余连通分量
3. 旧容量 C_old，新总容量 C_new = (总剩余块数)*24
4. Compact 旧内容
5. 若 UsedCount 对应「占用槽数」> C_new：
   - 超出部分从尾部槽依次取出，在 worldPos 生成 Pickable（爆出）
6. 若剩余 0 块：销毁 Vault（内容应已在步骤 5 清空或爆出）
7. 若剩余 1 个分量：同一 Vault Resize 到 C_new，锚点可能迁移（物品跟着迁）
8. 若剩余 ≥2 个分量（中间挖断）：
   - 按分量拆成多个 Vault（一分量保留旧 Guid，其余 NewGuid）
   - 物品按 **Q1**：紧密槽序切分——分量 i 分得容量 `Ci` 对应的前缀段
9. 再 Remove 本格 Entity；Unit.DropAllItems 必须是 no-op
```

**关键**：覆盖 / 绕过 `BlockEntityUtils.RemoveBlockEntity` 的默认 `DropAllItems`，否则缩容逻辑与宿主 Drop 会**双倍洒落**或顺序错乱。

### 5.4 邻接变化

`OnNeighborBlockChanged`：若邻格是储存单元增删，对受影响坐标 `RebuildAt`（可防抖同帧合并）。

## 7. UI

### 7.1 布局约束（仓库规则）

机器模态：**左储存库、右玩家背包**；勿用可见 Bevel 堆分区。参考 `IronFurnaceWidget` / 箱子面板骨架。

### 7.2 分页方案（推荐）

常量建议（可调）：

- `PageColumns = 6`，`PageRows = 4` → **每页 24 格**（恰好 1 单元视觉量）
- 或 `9×4 = 36` 等，取 **4 的倍数** 尺寸

控件：

- `StorageVaultWidget`：页码标签、上一页 / 下一页、当前页 `GridPanelWidget` 内 `InventorySlotWidget`
- 绑定：`AssignInventorySlot(unitInventory, page * pageSize + i)`
- 容量变化或 Compact 后：若当前页越界则夹到末页并 `RebuildPage()`
- 大页也可用 `LogisticsScrollPanelWidget` 作备选；需求写明「分页」，**第一期以页码为主**，滚动为可选增强
